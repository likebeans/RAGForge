## 1. 项目背景与动机

### 1.1 行业背景

药物研发是一个高度复杂、周期长、投入大的过程。从靶点发现、先导化合物筛选、临床前研究到临床试验，整个流程可能耗时 10-15 年，投入数十亿美元。在这个过程中，会产生海量的实验数据、文档资料、分析报告等，这些数据分散在不同的系统、文件夹和个人电脑中，缺乏统一的管理和有效的利用。

随着人工智能技术的发展，特别是大语言模型（LLM）和检索增强生成（RAG）技术的成熟，为药物研发数据的智能化管理和利用提供了新的可能性。通过构建一个统一的数据平台，结合 AI 技术，可以显著提升研发效率，加速知识积累和复用。

### 1.2 项目目标

本项目旨在构建一个**药研 AI 平台**，实现以下核心目标：

1. **数据集中管理**：将分散在各处的研发数据、文档统一存储和管理
2. **权限精细控制**：基于角色和项目的细粒度权限管理，确保数据安全
3. **智能检索与问答**：基于 RAG 技术，实现对研发文档的智能检索和问答
4. **流程标准化**：规范项目管理、实验记录、数据提交等流程
5. **数据资产积累**：将研发过程中的知识和经验转化为可复用的数据资产

### 1.3 业务价值

- **提升研发效率**：快速检索历史数据，避免重复实验，缩短研发周期
- **降低研发成本**：通过知识复用和经验共享，减少试错成本
- **保障数据安全**：细粒度权限控制，确保敏感数据不被泄露
- **支持决策分析**：通过数据可视化和智能分析，辅助管理层决策
- **满足合规要求**：符合 GLP/GCP、21 CFR Part 11 等法规要求

## 2. 当前问题与痛点

### 2.1 数据管理问题

#### 2.1.1 数据分散

- **问题描述**：研发数据分散在多个系统中，包括：
  - 实验室信息管理系统（LIMS）
  - 电子实验记录本（ELN）
  - 文档管理系统（DMS）
  - 个人电脑和共享文件夹
  - 第三方合作机构的系统

- **影响**：
  - 数据查找困难，耗费大量时间
  - 数据孤岛严重，难以进行跨项目分析
  - 数据版本混乱，容易使用过期数据

#### 2.1.2 缺乏标准化

- **问题描述**：
  - 不同项目组使用不同的数据格式和命名规范
  - 实验记录格式不统一，字段定义不一致
  - 缺乏统一的数据字典和元数据管理

- **影响**：
  - 数据难以整合和比对
  - 数据质量参差不齐
  - 数据复用困难

#### 2.1.3 检索效率低

- **问题描述**：
  - 依赖文件名和文件夹结构进行检索
  - 无法进行全文检索和语义检索
  - 缺乏智能推荐和关联分析

- **影响**：
  - 查找历史数据耗时长
  - 难以发现相关的实验和文档
  - 知识复用率低

### 2.2 权限管理问题

#### 2.2.1 权限粒度粗

- **问题描述**：
  - 现有系统通常只支持文件夹级别的权限控制
  - 无法实现基于项目、文档类型、敏感级别的细粒度控制
  - 缺乏动态权限管理机制

- **影响**：
  - 数据安全风险高
  - 权限管理复杂，容易出错
  - 难以满足合规要求

#### 2.2.2 审计追溯困难

- **问题描述**：
  - 缺乏完整的操作日志记录
  - 无法追溯数据的访问和修改历史
  - 审计报告生成困难

- **影响**：
  - 数据泄露难以追查
  - 不符合 FDA 等监管要求
  - 内部审计成本高

### 2.3 协作问题

#### 2.3.1 跨团队协作困难

- **问题描述**：
  - 不同团队使用不同的工具和流程
  - 缺乏统一的沟通和协作平台
  - 项目进度和状态不透明

- **影响**：
  - 沟通成本高，效率低
  - 项目延期风险大
  - 团队协作体验差

#### 2.3.2 知识传承困难

- **问题描述**：
  - 研发经验和知识主要依赖个人记忆
  - 人员流动导致知识流失
  - 缺乏系统化的知识管理

- **影响**：
  - 新人上手慢
  - 重复犯错
  - 组织知识资产流失

### 2.4 合规问题

#### 2.4.1 法规要求

- **GLP/GCP 要求**：
  - 数据完整性（ALCOA+ 原则）
  - 审计追踪
  - 电子签名

- **21 CFR Part 11 要求**：
  - 系统验证
  - 访问控制
  - 审计追踪
  - 电子签名和记录

#### 2.4.2 当前差距

- **问题描述**：
  - 现有系统不满足完整的合规要求
  - 缺乏系统化的验证流程
  - 审计追踪不完整

- **影响**：
  - 监管检查风险
  - 数据可信度受质疑
  - 可能影响药物审批

## 3. 用户与使用场景分析

### 3.1 目标用户

#### 3.1.1 研发人员

**角色特征**：
- 药物化学家、生物学家、药理学家等
- 日常进行实验设计、数据记录、结果分析
- 需要频繁查阅历史数据和文献

**核心需求**：
- 快速检索相关实验数据和文档
- 记录和管理实验过程
- 与团队成员共享数据
- 获取智能建议和推荐

**使用频率**：每天多次

#### 3.1.2 项目经理

**角色特征**：
- 负责项目整体规划和进度管理
- 需要了解项目状态和资源使用情况
- 协调跨团队协作

**核心需求**：
- 查看项目进度和里程碑
- 管理项目文档和数据
- 生成项目报告
- 进行资源分配和风险评估

**使用频率**：每天

#### 3.1.3 数据分析师

**角色特征**：
- 负责数据整合、分析和可视化
- 需要访问多个项目的数据
- 进行统计分析和建模

**核心需求**：
- 批量导出和处理数据
- 进行跨项目数据分析
- 创建数据可视化报表
- 使用 API 集成分析工具

**使用频率**：每周多次

#### 3.1.4 质量管理人员

**角色特征**：
- 负责质量控制和合规审查
- 需要审计数据和操作记录
- 确保符合法规要求

**核心需求**：
- 查看审计日志
- 生成合规报告
- 管理电子签名
- 进行数据完整性检查

**使用频率**：每周

#### 3.1.5 管理层

**角色特征**：
- 高层管理人员，关注整体研发进展
- 需要做出战略决策
- 关注投资回报和风险

**核心需求**：
- 查看研发管线和项目状态
- 获取数据驱动的决策支持
- 了解资源使用和成本
- 进行项目优先级排序

**使用频率**：每周或每月

### 3.2 核心使用场景

#### 3.2.1 场景 1：智能文档检索

**场景描述**：
研发人员需要查找某个化合物的历史实验数据，但不记得具体的文档名称和位置。

**用户操作流程**：
1. 在搜索框输入自然语言查询："化合物 XYZ-123 的体外活性数据"
2. 系统基于 RAG 技术进行语义检索
3. 返回相关文档列表，按相关性排序
4. 用户点击查看文档详情
5. 系统显示文档内容，并高亮相关段落

**系统能力要求**：
- 全文索引和向量化
- 语义理解和匹配
- 权限过滤（只返回用户有权访问的文档）
- 快速响应（< 2 秒）

#### 3.2.2 场景 2：智能问答

**场景描述**：
研发人员想了解某个靶点的研究进展，但不想逐个阅读所有文档。

**用户操作流程**：
1. 在问答界面输入问题："EGFR 靶点的最新研究进展是什么？"
2. 系统检索相关文档片段
3. 使用 LLM 生成综合性回答
4. 显示答案，并附上引用来源
5. 用户可以继续追问或查看原始文档

**系统能力要求**：
- RAG 技术集成
- 多文档信息整合
- 引用溯源
- 上下文理解（支持多轮对话）

#### 3.2.3 场景 3：项目数据管理

**场景描述**：
项目经理需要管理项目的基本信息、文档和进度。

**用户操作流程**：
1. 进入项目管理页面
2. 创建新项目，填写项目信息（名称、阶段、适应症等）
3. 上传项目相关文档
4. 设置项目成员和权限
5. 更新项目状态和里程碑
6. 查看项目仪表板和报告

**系统能力要求**：
- 项目 CRUD 操作
- 文档上传和关联
- 权限管理
- 状态跟踪
- 数据可视化

#### 3.2.4 场景 4：实验数据记录

**场景描述**：
研发人员完成实验后，需要记录实验过程和结果。

**用户操作流程**：
1. 创建新的实验记录
2. 填写实验信息（目的、方法、材料等）
3. 上传实验数据文件（Excel、图片等）
4. 记录实验结果和结论
5. 关联到相应的项目
6. 提交审核（如需要）

**系统能力要求**：
- 结构化数据录入
- 文件上传和预览
- 数据验证
- 版本控制
- 审批流程

#### 3.2.5 场景 5：数据导出与分析

**场景描述**：
数据分析师需要导出多个项目的数据进行统计分析。

**用户操作流程**：
1. 进入数据导出页面
2. 选择项目和数据类型
3. 设置筛选条件和时间范围
4. 选择导出格式（Excel、CSV、JSON）
5. 下载数据文件
6. 使用外部工具进行分析

**系统能力要求**：
- 灵活的数据筛选
- 多格式导出
- 批量操作
- API 支持

#### 3.2.6 场景 6：权限审批

**场景描述**：
用户需要访问某个受限项目的数据，需要申请权限。

**用户操作流程**：
1. 尝试访问受限文档，系统提示无权限
2. 点击"申请权限"按钮
3. 填写申请理由
4. 提交申请
5. 项目负责人收到通知，审批申请
6. 用户收到审批结果通知
7. 获得权限后可以访问文档

**系统能力要求**：
- 权限申请流程
- 审批工作流
- 消息通知
- 权限时效管理

#### 3.2.7 场景 7：审计追溯

**场景描述**：
质量管理人员需要审计某个文档的访问和修改历史。

**用户操作流程**：
1. 进入审计日志页面
2. 输入文档 ID 或名称
3. 设置时间范围
4. 查看操作记录列表（访问、修改、下载等）
5. 导出审计报告
6. 分析异常操作

**系统能力要求**：
- 完整的操作日志记录
- 灵活的查询和筛选
- 报告生成
- 异常检测

### 3.3 用户旅程地图

#### 3.3.1 新用户首次使用

1. **注册/邀请**：管理员创建账号并分配初始角色
2. **登录**：使用企业邮箱和密码登录
3. **引导教程**：系统展示核心功能介绍
4. **加入项目**：被添加到相关项目组
5. **浏览文档**：查看项目文档列表
6. **首次检索**：尝试搜索功能
7. **上传文档**：上传第一个文档
8. **体验问答**：使用智能问答功能

#### 3.3.2 日常使用流程

1. **登录系统**
2. **查看待办事项**：审批、通知等
3. **检索文档**：查找需要的资料
4. **记录实验**：上传新的实验数据
5. **协作交流**：与团队成员讨论
6. **更新项目**：更新项目状态
7. **退出系统**

| 全文检索 | P0 | 核心价值功能 |
| 项目管理 | P1 | 重要的组织功能 |

| 审计日志 | P1 | 合规要求 |
| 数据可视化 | P2 | 提升用户体验 |
| 移动端支持 | P2 | 便利性功能 |
| 高级分析 | P3 | 增值功能 |
