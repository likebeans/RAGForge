## 6. 核心功能详细设计

### 6.1 数据接入与管理

#### 6.1.1 文档上传

**功能描述**：
支持多种方式上传文档到平台，包括单文件上传、批量上传、拖拽上传等。

**交互流程**：
1. 用户点击"上传文档"按钮
2. 选择上传方式（单文件/批量/拖拽）
3. 选择文件（支持多选）
4. 填写文档元数据：
   - 标题（可选，默认为文件名）
   - 描述（可选）
   - 标签（可选，支持多标签）
   - 分类（必选，从预定义分类中选择）
   - 关联项目（可选）
   - 敏感级别（必选：公开/内部/机密）
5. 点击"开始上传"
6. 显示上传进度
7. 上传完成后，自动触发：
   - 文件存储到 MinIO
   - 元数据保存到 PostgreSQL
   - 文本提取和向量化（异步任务）
   - 权限初始化

**支持的文件类型**：
- 文档：PDF, Word (.docx), Excel (.xlsx), PowerPoint (.pptx), TXT, Markdown
- 图片：JPG, PNG, GIF, BMP, TIFF
- 数据：CSV, JSON, XML
- 其他：ZIP（自动解压）

**技术实现**：
- 前端：使用 `react-dropzone` 实现拖拽上传
- 后端：FastAPI 的 `UploadFile` 处理文件流
- 存储：MinIO SDK 上传到对象存储
- 异步任务：Celery 任务处理文本提取和向量化

**限制与验证**：
- 单文件大小限制：500 MB
- 批量上传数量限制：50 个文件
- 文件名长度限制：255 字符
- 文件类型白名单验证
- 病毒扫描（可选）

#### 6.1.2 文档预览

**功能描述**：
在浏览器中直接预览文档内容，无需下载。

**支持的预览方式**：
- **PDF**：使用 PDF.js 在线预览
- **Office 文档**：转换为 PDF 后预览，或使用 Office Online
- **图片**：直接显示
- **文本文件**：语法高亮显示
- **其他格式**：提供下载链接

**交互流程**：
1. 用户点击文档名称或"预览"按钮
2. 系统检查用户权限
3. 打开预览窗口
4. 加载文档内容
5. 提供工具栏：缩放、翻页、下载、打印等

**技术实现**：
- 前端：React PDF Viewer / Office Web Viewer
- 后端：文档转换服务（LibreOffice / Pandoc）
- 缓存：预览文件缓存到 Redis

#### 6.1.3 文档版本控制

**功能描述**：
记录文档的所有历史版本，支持版本比较和回滚。

**版本管理规则**：
- 每次上传同名文件时，自动创建新版本
- 版本号格式：v1.0, v1.1, v2.0（主版本.次版本）
- 保留所有历史版本（可配置保留策略）
- 支持版本标签（如：草稿、审核中、已发布）

**交互流程**：
1. 用户查看文档详情
2. 点击"版本历史"标签
3. 显示版本列表：版本号、上传时间、上传人、文件大小、备注
4. 用户可以：
   - 查看任意版本
   - 下载任意版本
   - 比较两个版本（文本文件）
   - 恢复到某个版本

**技术实现**：
- 存储：每个版本独立存储在 MinIO
- 数据库：记录版本元数据和关系
- 版本比较：使用 diff 算法

#### 6.1.4 文档分类与标签

**功能描述**：
通过分类和标签组织文档，便于检索和管理。

**分类体系**（预定义，可扩展）：
- 研究报告
  - 靶点研究
  - 化合物筛选
  - 药效学研究
  - 药代动力学研究
  - 毒理学研究
- 实验记录
  - 体外实验
  - 体内实验
  - 临床前研究
- 临床文档
  - 临床方案
  - 研究者手册
  - 知情同意书
  - 临床报告
- 监管文档
  - IND/NDA 申报资料
  - 审评意见
  - 补充资料
- 项目管理
  - 项目计划
  - 会议纪要
  - 决策记录
- 其他

**标签系统**（自由标签）：
- 用户可以自定义标签
- 支持标签自动补全
- 热门标签推荐
- 标签云展示

**交互流程**：
1. 上传或编辑文档时选择分类
2. 添加标签（输入或选择）
3. 保存后，文档归入相应分类
4. 可通过分类和标签筛选文档

### 6.2 数据标准化

#### 6.2.1 数据字典管理

**功能描述**：
定义和管理平台中使用的标准术语、枚举值和数据规范。

**数据字典类型**：
- **药物类型**：小分子、大分子、细胞治疗、基因治疗等
- **适应症**：肿瘤、心血管、神经系统、代谢疾病等
- **研发阶段**：靶点发现、先导化合物、临床前、临床 I/II/III 期、上市
- **项目状态**：启动、进行中、暂停、终止、完成
- **实验类型**：体外活性、体内药效、ADME、毒理等
- **文档分类**：见上文
- **敏感级别**：公开、内部、机密、绝密

**管理功能**：
- 查看数据字典列表
- 添加/编辑/删除字典项
- 设置字典项的显示顺序
- 启用/禁用字典项
- 导入/导出数据字典

**技术实现**：
- 数据库表：`dict_items`（字典类型、字典值、显示名称、排序、状态）
- API：CRUD 接口
- 前端：管理界面（仅管理员可访问）

#### 6.2.2 元数据提取

**功能描述**：
自动从上传的文档中提取元数据，减少手工输入。

**提取内容**：
- **文件元数据**：文件名、大小、类型、创建/修改时间
- **文档属性**：标题、作者、关键词（从 Office 文档属性）
- **文本内容**：提取全文用于检索和向量化
- **结构化数据**：从 Excel 中提取表格数据

**提取方法**：
- **PDF**：使用 PyPDF2 / pdfplumber 提取文本和元数据
- **Office 文档**：使用 python-docx / openpyxl 提取
- **图片**：使用 OCR（Tesseract / Paddle OCR）提取文字
- **其他**：使用 Apache Tika 统一处理

**技术实现**：
- 异步任务：Celery 任务在后台处理
- 任务队列：Redis
- 结果存储：PostgreSQL（元数据）+ Milvus（向量）

#### 6.2.3 数据验证

**功能描述**：
在数据录入和导入时进行验证，确保数据质量。

**验证规则**：
- **必填字段验证**：项目名称、药物类型、研发阶段等
- **格式验证**：日期格式、邮箱格式、数值范围等
- **唯一性验证**：项目编号、文档编号等
- **引用完整性验证**：关联的项目、用户等必须存在
- **业务规则验证**：如阶段顺序、状态转换规则等

**验证时机**：
- 前端实时验证（用户体验）
- 后端提交验证（数据安全）
- 批量导入验证（数据质量）

**错误处理**：
- 显示清晰的错误信息
- 标记错误字段
- 提供修正建议

### 6.3 项目与实验管理

#### 6.3.1 项目信息管理

**功能描述**：
管理药物研发项目的基本信息和状态。

**数据模型**（基于现有 `project.py`）：
```python
class Project:
    id: int
    project_code: str  # 项目编号（唯一）
    project_name: str  # 项目名称
    drug_type: str  # 药物类型（数据字典）
    indication: str  # 适应症（数据字典）
    target: str  # 靶点
    stage: str  # 研发阶段（数据字典）
    status: str  # 项目状态（数据字典）
    priority: str  # 优先级（高/中/低）
    start_date: date  # 开始日期
    end_date: date  # 结束日期（可选）
    description: str  # 项目描述
    owner_id: int  # 项目负责人
    created_at: datetime
    updated_at: datetime
```

**CRUD 操作**：
- **创建项目**：填写项目信息表单，提交后创建
- **查看项目**：列表页（支持筛选和排序）+ 详情页
- **编辑项目**：修改项目信息（权限控制）
- **删除项目**：软删除（标记为已删除，不物理删除）

**项目筛选**（参考 `PROJECT_FILTER_DESIGN.md`）：
- 按药物类型筛选
- 按适应症筛选
- 按靶点筛选
- 按研发阶段筛选
- 按项目状态筛选
- 按优先级筛选
- 按时间范围筛选
- 组合筛选（多条件 AND/OR）

**项目导入导出**：
- 导入：Excel 模板导入项目数据
- 导出：导出项目列表为 Excel

**技术实现**：
- API：`/api/projects`（CRUD + 筛选）
- 前端：项目列表页 + 项目详情页 + 项目表单
- 数据库：`drug_projects` 表

### 6.4 数据查询、分析与可视化

#### 6.4.1 全文检索

**功能描述**：
基于关键词的全文检索，快速查找文档。

**检索范围**：
- 文档标题
- 文档描述
- 文档内容（全文）
- 标签
- 文件名

**检索功能**：
- **简单检索**：输入关键词，返回匹配结果
- **高级检索**：
  - 精确匹配（"关键词"）
  - 布尔运算（AND, OR, NOT）
  - 通配符（*, ?）
  - 字段检索（title:关键词）
- **筛选器**：
  - 按文档类型筛选
  - 按项目筛选
  - 按上传时间筛选
  - 按上传人筛选
- **排序**：
  - 相关性排序（默认）
  - 时间排序
  - 文件大小排序

**搜索结果展示**：
- 文档列表
- 高亮显示匹配的关键词
- 显示文档摘要
- 分页显示

**技术实现**：
- 搜索引擎：Elasticsearch（推荐）或 PostgreSQL 全文搜索
- 索引：文档上传时自动索引
- API：`/api/search`
- 前端：搜索框 + 结果列表

#### 6.4.2 语义检索

**功能描述**：
基于语义理解的检索，支持自然语言查询。

**工作原理**：
1. 用户输入自然语言查询（如："EGFR 抑制剂的药效学数据"）
2. 将查询文本向量化（Embedding）
3. 在向量数据库（Milvus）中进行相似度检索
4. 返回语义相关的文档
5. 结合全文检索结果，综合排序

**技术实现**：
- Embedding 模型：OpenAI text-embedding-3 / BGE-large / M3E
- 向量数据库：Milvus
- 相似度算法：余弦相似度
- API：`/api/semantic-search`

**优化策略**：
- 混合检索：结合全文检索和语义检索
- 重排序（Rerank）：使用 Rerank 模型优化结果排序
- 缓存：热门查询结果缓存

#### 6.4.3 智能问答（RAG）

**功能描述**：
基于 RAG（Retrieval-Augmented Generation）技术，回答用户关于研发数据的问题。

**工作流程**：
1. **用户提问**：输入自然语言问题
2. **检索相关文档**：
   - 将问题向量化
   - 在 Milvus 中检索 Top-K 相关文档片段
   - 权限过滤（只检索用户有权访问的文档）
3. **构建提示词**：
   - 将检索到的文档片段作为上下文
   - 结合用户问题构建 Prompt
4. **调用 LLM**：
   - 发送 Prompt 到 LLM API
   - 获取生成的答案
5. **返回结果**：
   - 显示答案
   - 显示引用来源（文档名称、片段位置）
   - 支持追问（多轮对话）

**Prompt 模板**：
```
你是一个药物研发领域的专家助手。请基于以下文档内容回答用户的问题。

文档内容：
---
{document_chunks}
---

用户问题：{user_question}

请提供准确、专业的回答，并在回答中标注信息来源。如果文档中没有相关信息，请明确说明。
```

**多轮对话**：
- 保存对话历史
- 将历史对话作为上下文
- 支持追问和澄清

**引用溯源**：
- 在答案中标注引用的文档
- 点击引用可跳转到原文档
- 高亮显示引用的段落

**技术实现**：
- RAG 框架：LangChain / LlamaIndex
- LLM：OpenAI GPT-4 / Claude / 本地模型
- 向量数据库：Milvus
- 会话管理：Redis（存储对话历史）
- API：`/api/chat`

**优化策略**：
- **文档分块**：将长文档切分为小块（chunk），提高检索精度
- **混合检索**：结合关键词检索和向量检索
- **重排序**：使用 Rerank 模型优化检索结果
- **Prompt 优化**：针对药物研发领域优化 Prompt
- **流式输出**：使用 SSE（Server-Sent Events）实现流式返回

### 6.5 权限、审计与合规

#### 6.5.1 权限模型

**RBAC（基于角色的访问控制）**：

**系统角色**：
- **超级管理员**：所有权限
- **管理员**：用户管理、系统配置
- **项目经理**：项目管理、成员管理
- **研发人员**：文档上传、实验记录
- **数据分析师**：数据查询、导出
- **质量管理员**：审计日志、合规检查
- **访客**：只读权限

**权限类型**：
- **读取（Read）**：查看文档/项目
- **写入（Write）**：创建/编辑文档/项目
- **删除（Delete）**：删除文档/项目
- **管理（Manage）**：权限管理、成员管理
- **审计（Audit）**：查看审计日志

**ABAC（基于属性的访问控制）**：

**属性维度**：
- **用户属性**：角色、部门、级别
- **资源属性**：项目、分类、敏感级别
- **环境属性**：时间、IP 地址、设备

**权限规则示例**：
```
IF 用户.角色 == "研发人员" 
   AND 文档.项目 IN 用户.项目列表
   AND 文档.敏感级别 <= 用户.级别
THEN 允许读取
```

**技术实现**：
- 权限表：`permissions`（subject_type, subject_id, resource_type, resource_id, permission_type）
- 权限检查：装饰器 `@require_permission`
- 权限缓存：Redis 缓存用户权限

#### 6.5.2 权限申请与审批

**功能描述**：
用户可以申请访问受限资源，由资源所有者审批。

**申请流程**：
1. 用户尝试访问受限资源，系统提示无权限
2. 用户点击"申请权限"
3. 填写申请表单：
   - 申请理由
   - 申请的权限类型（读/写）
   - 申请期限（临时/永久）
4. 提交申请
5. 系统通知资源所有者或项目负责人
6. 审批人审批（同意/拒绝）
7. 系统通知申请人结果
8. 如同意，自动授予权限

**技术实现**：
- 数据库：`permission_requests` 表
- 通知：邮件 + 站内消息
- API：`/api/permission-requests`

#### 6.5.3 审计日志

**功能描述**：
记录所有用户操作，用于审计和追溯。

**记录的操作**：

#### 6.5.4 数据完整性

**ALCOA+ 原则**：
- **Attributable（可归属）**：记录操作人
- **Legible（可读）**：数据清晰可读
- **Contemporaneous（同步）**：实时记录
- **Original（原始）**：保留原始数据
- **Accurate（准确）**：数据准确无误
- **Complete（完整）**：数据完整
- **Consistent（一致）**：数据一致
- **Enduring（持久）**：数据持久保存
- **Available（可用）**：数据可访问

**实现措施**：
- 所有操作记录审计日志（Attributable, Contemporaneous）
- 版本控制保留历史（Original）
- 数据验证确保准确性（Accurate）
- 必填字段确保完整性（Complete）
- 事务保证一致性（Consistent）
- 备份保证持久性（Enduring）
- 权限控制保证可用性（Available）

#### 6.5.5 电子签名（可选）

**功能描述**：
对关键操作和文档进行电子签名，满足 21 CFR Part 11 要求。

**签名场景**：
- 实验记录提交
- 文档发布
- 审批操作
- 数据导出

**签名流程**：
1. 用户执行需要签名的操作

5. 记录签名信息（签名人、签名时间、签名内容）
6. 完成操作

**技术实现**：
- 签名方式：密码签名 / 数字证书
- 签名记录：`signatures` 表
- 签名验证：哈希验证
